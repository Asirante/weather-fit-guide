🚨 하이브리드 배포 가이드 (V3)

학교 AWS 계정의 보안 정책(Explicit Deny)을 우회하기 위해,
[인프라 수동 갱신]과 [코드 자동 배포]를 분리하는 하이브리드 전략을 채택했습니다.

상황 A. 파이썬 코드 / 로직을 수정했을 때

예: lambda_function.py, 프론트엔드 코드 수정

GitHub Actions가 100% 자동 처리합니다.
로컬에서 코드를 수정한 뒤 GitHub에 `git push`만 하세요. AWS 콘솔이나 CloudShell에 들어갈 필요가 전혀 없습니다.

  • 백엔드/데이터: 코드가 압축되어 AWS 람다로 자동 업데이트 (update-function-code)
  • 프론트엔드 (최혁 님): 작업 예정 (아직 CI/CD 적용 전입니다)

상황 B. 인프라 설정 (template.yaml)을 수정했을 때

예: 메모리 변경, 시간 변경, 새 DB 추가, 환경 변수 추가

인프라 설정 변경은 깃허브 액션이 자동으로 처리하지 못합니다! (권한 없음)
반드시 "해당 파트의 담당자"가 본인의 CloudShell에서 수동으로 배포해야 합니다.

수정된 폴더 담당자 CloudShell 갱신 명령어 (그대로 복사)
infra/ PM (-2-) cd infra sam build sam deploy \ --stack-name inhatc-team2-infra \ --region us-east-1 \ --no-confirm-changeset \ --capabilities CAPABILITY_IAM \ --s3-bucket inhatc-team2-5-raw-data
backend/ 고원영 (-1-) cd backend sam build sam deploy \ --stack-name inhatc-team2-1-backend \ --region us-east-1 \ --no-confirm-changeset \ --capabilities CAPABILITY_IAM \ --s3-bucket inhatc-team2-5-raw-data
data_pipeline/ 김호건 (-5-) cd data_pipeline sam build sam deploy \ --stack-name inhatc-team2-5-pipeline \ --region us-east-1 \ --no-confirm-changeset \ --capabilities CAPABILITY_IAM \ --s3-bucket inhatc-team2-5-raw-data \ --parameter-overrides WeatherApiKey="본인기상청키입력"
real_time/ 김호건 (-5-) cd real_time sam build sam deploy \ --stack-name inhatc-team2-5-realtime \ --region us-east-1 \ --no-confirm-changeset \ --capabilities CAPABILITY_IAM \ --s3-bucket inhatc-team2-5-raw-data \ --parameter-overrides WeatherApiKey="본인기상청키입력" AirApiKey="본인에어코리아키입력"

절대 주의사항

  • 다른 팀원의 인프라 스택을 대신 배포하려 하지 마세요. 학교 계정 정책상 AccessDenied 에러가 발생합니다.
  • 환경변수 추가가 필요하다면 template.yaml 수정 후 반드시 해당 팀원이 CloudShell에서 배포해야 반영됩니다.
  • AWS 콘솔(웹사이트)에서 마우스로 람다 설정을 변경하지 마세요. 다음 배포 시 덮어씌워져 초기화됩니다.
