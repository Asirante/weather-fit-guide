CI/CD 자동 배포 파이프라인 구축이 완료되었습니다!
이제부터 AWS 콘솔(웹사이트)에서 직접 코드를 짜거나 인프라를 수정하는 행위를 전면 금지합니다.
기존에 만들어둔 모든 작업물을 GitHub 레포지토리로 이관하고 동기화하는 작업을 즉시 진행해 주세요.

1

기존 작업물 소스 코드 Git 이관

현재 AWS 콘솔이나 개인 로컬에만 존재하는 코드들을 깃허브의 알맞은 폴더로 옮겨주세요.

⚙️

백엔드 (고원영 님)

현재 람다(Lambda) 콘솔에서 직접 테스트했던 FastAPI 및 추천 API 파이썬 코드 전체를 backend/ 폴더 안으로 가져와서 커밋해 주세요.

🗄️

데이터 엔지니어 (김호건 님)

기상청/에어코리아 API를 호출하고 데이터를 수집/정제하는 파이썬 코드를 data_pipeline/ 폴더로 가져와서 커밋해 주세요.

🖥️

프론트엔드 (최혁 님)

더미 데이터로 구성해 둔 Vue.js 프로토타입 소스 코드를 frontend/ 폴더 안에 넣고 커밋해 주세요.

2

AWS 인프라 설정 SAM (template.yaml) 동기화

단순 코드뿐만 아니라, 콘솔에서 마우스로 설정한 인프라 설정(IaC)도 모두 문서화해야 합니다. 이를 맞추지 않으면 자동 배포 시 에러가 발생하거나 설정이 날아갈 수 있습니다.

To-Do: 본인이 AWS 콘솔에서 설정한 람다 메모리 크기, 타임아웃, 생성한 환경 변수 등을 각 파트 폴더에 있는 template.yaml 파일에 코드로 똑같이 채워 넣어주세요!
AWS 인프라 리소스 관리 방법 (어떻게 만들어야 하나요?)
Lambda, EventBridge 각 폴더의 template.yaml에 정의하면 배포 시 자동 반영됩니다.
S3, DynamoDB, CloudFront infra/template.yaml 내에 코드로 인프라를 정의하여 배포합니다.
Athena (데이터 테이블) infra/athena_ddl/ 폴더 안에 SQL 파일로 저장하여 관리합니다.
Amazon Bedrock (예외) Bedrock 모델 접근 권한(Model Access)은 코드(SAM)로 제어할 수 없습니다. 이 부분만 예외적으로 콘솔에 들어가 수동으로 권한을 요청합니다.

💡 새로운 작업 프로세스 요약

1. 로컬 환경에서 코드 및 template.yaml 수정
2. feature/* 브랜치 생성 후 GitHub에 Push
3. PR 생성 후 팀원 승인을 받아 develop 브랜치로 병합 (자동 배포)
