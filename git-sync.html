최종 확정 배포 프로세스 (V4)

🚨 하이브리드 배포 가이드

학교 AWS 계정의 강력한 보안 정책(Lambda 자동 생성 차단)에 맞춰,
[인프라 웹 콘솔 수동 갱신]과 [파이썬 코드 자동 배포]를 철저히 분리합니다.

상황 A. 파이썬 코드 / 로직을 수정했을 때

예: lambda_function.py 등 파이썬 파일 수정 시

GitHub Actions가 100% 자동 처리합니다.
로컬에서 코드를 수정한 뒤 GitHub에 `git push`만 하세요. AWS 콘솔이나 CloudShell에 들어갈 필요가 전혀 없습니다.

  • 백엔드/데이터: 코드가 압축되어 AWS 람다로 자동 업데이트 (update-function-code)
  • 프론트엔드 (최혁 님): 작업 예정 (아직 CI/CD 적용 전입니다)

상황 B. 인프라 설정(환경변수, 트리거 시간 등)을 수정해야 할 때

예: 새로운 API 키 추가, 수집 스케줄 변경, 람다 메모리 증설

학교 계정 정책상 람다 제어(CloudFormation) 권한이 명시적 거부(Deny)로 차단되어 있습니다.
팀원들은 CloudShell에서 sam deploy 명령어를 절대 사용하지 마세요. 에러가 발생합니다.

AWS 웹 콘솔에서 직접(수동) 수정하세요!
  • 환경 변수 추가: AWS Lambda > 해당 함수 > 구성 > 환경 변수
  • 트리거 시간 변경: EventBridge 메뉴에서 직접 규칙(Rule) 수정
  • template.yaml 파일은 변경 내역 추적용(문서용)으로만 수정해서 Push 하세요. 배포에 반영되지 않습니다.

결론: 우리의 확실한 작업 방식

  • 1

    람다 껍데기와 설정(환경변수 등)은 AWS 웹 콘솔에서 마우스로 만듭니다.

  • 2

    그 안의 파이썬 코드(.py)는 로컬에서 짜고 깃허브에 푸시합니다.

  • 3

    나머지는 깃허브 액션이 1분 안에 람다 코드를 갈아끼워주며 자동으로 완성됩니다.
